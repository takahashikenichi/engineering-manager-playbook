# 第4章：採用と評価 - 優秀な人材の獲得と評価制度

## はじめに

「採用と評価は、マネージャーが行う仕事で最も重要なものの一つである」
これは私の上司や同僚のマネージャーから、事ある毎に交わされた言葉です。

一人の優秀なエンジニアの採用は、開発チームの生産性を飛躍的に向上させますが、逆に不適切なメンバーの採用は、チーム全体のパフォーマンスを著しく低下させ、その修復には膨大なコストと時間がかかります。また、評価についても同じ事が言えます。公正で透明な評価制度は、優秀な人材を維持し、成長を促進する鍵となります。一方で、評価の仕組みが機能していないと、メンバーの成長機会を奪い、信頼関係を損なうことになります。

本章では、採用と評価まで、人材マネジメントの核心部分について考えていきます。テンプレートやフレームワークも紹介いたしますが、それらを機械的に当てはめることはせず、「このチームで働きたいと思ってもらえるか」「このメンバーが成長するために何ができるか」という問いに真摯に向き合うことが大切であると考えます。

## 4.1 採用の重要性と影響

### 採用の乗数効果

スティーブ・ジョブズは「A級の人材はA級の人材を呼ぶが、B級の人材はC級の人材を採用する」と言いました。この言葉はやや極端ではありますが、採用の連鎖的な影響を端的に表しているとも言えます。

技術力は申し分無いのに、コミュニケーションに難があるエンジニアを採用した、ということを考えてみましょう。会社のフェーズによっては、またはチームの状況によっては機能することもあるかもしれませんが、多くの場合は失敗する未来が見えるのではないでしょうか。攻撃的な会話はメンバーのコミュニケーションを萎縮させ、チーム全体の生産性を落とす結果にもなります。

逆に、技術力はまあまあでも学習意欲が高くチームを盛り上げてくれる人を採用した場合。チーム内での知識共有が活発になり、結果として全員のスキルが上がる、なんてことも大いに想像できるかと思います。

**優秀な採用がもたらす効果：**
- チーム全体のレベル向上（優秀なエンジニアは周囲の生産性も引き上げる）
- イノベーションの促進（新しい視点が新しいアイデアを生む）
- 学習文化の強化（あの人のようになりたい、という良い競争心）
- 採用ブランドの向上（著名で実力のあるエンジニアの存在）
- ネットワーク効果（優秀な人は優秀な知り合いを紹介してくれる）

**不適切な採用のコスト：**
- 生産性の低下（チーム全体の25-30%とも言われています）
- モラルへの悪影響（「なぜあの人が採用されたのか」という不信感）
- マネジメント時間の増大（本来必要のないマネジメントによって他の業務時間を奪われます）
- 再採用コスト（採用人件費などを考慮すると年収の1.5-2倍は損失になります）
- チーム文化の劣化（一度壊れた文化の修復は本当に大変です）と

### 採用におけるEMの役割

EMの仕事は多岐にわたりますが、この章の最初で紹介したように、採用はEMが行う仕事で最も重要なものの一つです。今日採用した人は3年後もそのチームにいるとしたら。その人の存在が文化を形作り、プロダクトの品質を左右し続けるのです。EMはその採用において責務を追うことになります。

EMが採用においてどのように関わるか、については、大きく3つのレベルに分けられます。

**戦略レベル：**
どのような開発をするのか、そのためにどのようなチームを作りたいのか。
- 人員計画の策定（半年〜数年）
- 採用基準の定義（どのような人をいま求めるのか）
- 採用ブランディング（中長期的にどのようなチームを作るのか・そのために何をブランディングするか）
- 多様性戦略（広い母集団から変化に強いチームを作る）

**戦術レベル：**
採用プロセスそのものの設計。
- ジョブディスクリプション作成（具体的にどのような仕事を任せたいか）
- 面接プロセス設計（人材を正しく見極めるための仕組み）
- 面接官のトレーニング（適切な判断ができるようにするため）
- 最終意思決定（最低限必要な要素、採用できる幅、許容できる幅）

**実行レベル：**
実際に採用を進めるプロセスの実施。
- スクリーニング（多くの候補者から条件に足る人を絞込む）
- 技術面接の実施（必要な技術力があるとみなせるだけの技術面接）
- 候補者体験の管理（親しみと信頼の両軸を採用まで一貫して）
- オファー交渉（年収テーブルと見立てたスキルの調整・競合他社との奪い合い）

## 4.2 人員計画と採用戦略

### 人員計画の策定
「人が足りません」と言うのは簡単ですが、「どのような人が何人足りないのか」「なぜ足りないのか」「いつまで必要なのか」を的確に説明できないと、採用活動に説得力が出ません。

前章では採用の関わり方について3つのレベルに分類しましたが、まずは計画を立てる必要があります。以下のステップで考えると良いでしょう。

計画フレームワーク：
1. ビジネス目標の理解
まず、この先半年〜1年で何を達成しなければならないのかを明確にします。新機能の開発、技術的負債の解消、スケーラビリティ対応など、具体的な目標を洗い出します。

2. 必要スキルの特定
目標を達成するために、どんなスキルセットが必要かを考えます。技術スタック、ドメイン知識、経験レベルの分布。「フロントエンドエンジニアが2人必要」ではなく、「Reactの経験があり、アクセシビリティの知識があるミドルレベルのエンジニアが2人必要」といった、具体的なスキルが判定できる解像度で設定します。

3. タイムラインの設定
採用には時間がかかります。求人を出してから入社までの平均リードタイムは3〜6ヶ月。ここにオンボーディング期間（通常1〜3ヶ月）を加えると、「来月からフル稼働してほしい」は現実的な計画ではありません。プロジェクトのマイルストーンから逆算して、いつまでに採用を完了すべきかを計画します。

4. 予算の確保
給与レンジ、採用コスト（エージェントへの手数料、求人広告費など）、採用してから実務に入るまでのトレーニング費用など、これらを具体的な数字にします。この具体的な数値から「何人必要か」の議論が「いくら必要か」という具体的な話に変わります。

### 採用チャネルの最適化 - 効果的な採用経路の選択

エンジニアを採用するにあたっては複数の経路があります。これを採用チャネルと呼びます。2026年現在での各チャネルの実態は以下のようになります。

**リファラル（社員紹介）**

「優秀な人は優秀な人を知っている」というのは前段でもご紹介したとおりで、獲得した人材が優秀であることを求めるのであればこれがもっとも確実です。しかし、リファラルを進めるには条件が揃っている必要はあります。普段から「今このような人材を探しています」という情報を共有することが重要です。また、メンバーが「自分の会社になら知り合いを入れてもいい」と思えるような環境である必要があります。ただ「紹介してくれたら3万円」といった制度だけ作っても全く機能しません。

**直接スカウト**

GitHubのコードやSNSの発信を見て「この人は素晴らしい」という人にDMを送ります。返信率は10%を切りますが、タイミングが合えば（その方がたまたま転職を検討しているなどしていれば）非常に優秀な人材を確保することができます。その際、メッセージはその人のコードの何が素晴らしいか、具体的に書いて興味を惹かれるものでないといけません。定型文は絶対に✕です。

**採用エージェント**

採用人件費の外注化、という観点では必要なチャネルではありますが、費用対効果という点では難しいものがあります。年収の35%程度、採用難易度が高い場合は50%以上の手数料がかかります。半年以内の退職は手数料の支払いがない、という契約が多いですが、すぐに退職されると大きな損失になります。しかし「3ヶ月後までに3人採用しないとプロジェクトが回らない」といった緊急時には、採用のテーブルに乗る母数が増える、という観点では頼りがいがあります。ここではエンジニア採用に強い専門エージェントを選ぶことが活用の鍵になります。

**求人サイト・ダイレクトリクルーティング**

WantedlyやGreen、Bizreachなどは、転職したい本人が直接企業と接点を持てる仕組みとなっており、直接スカウトなどに比べると求職者とのコンタクトを多く取れる利点があります。一方で、応募はあってもスクリーニングが大変です。100人応募があっても、面接したいと思える人は5人程度です。しかし、その5人の中から1人採用できれば成功と言えるでしょう。

**テックイベント・勉強会**

採用広報という考え方では、イベントや勉強会を使うのは長期的な効果はあります。メンバーが登壇したり、会社がスポンサーしたりすることで認知度を上げ、「あの会社の○○さんは技術力が高い」と思ってもらえれば、リファラル採用など他の採用チャネルへの良い影響をだすことができます。可能であればイベントの場で連絡先を交換し、定期的に連絡できる仕組みを作るとよいでしょう。ただし、イベントの際は採用目的を前面に出さず、純粋に技術交流することが重要です。

### 採用ブランディング - 小さい企業でも知名度を積み重ねる

GoogleやMercariのようなブランド力があれば採用もスムーズに行きますが、小さな会社だからこそできるブランディングもあります。

**技術ブログの定期的な執筆**

エンジニアは最新の技術動向を常に追いかけますが、必ずしも最新の情報である必要はありません。現場に近いリアルな話や、思いもつかないような失敗談、あるいはオンボーディングで淡々と進めたこと、などでも構いません。重要なのは「エンジニアが興味を引くトピックである」こと、そしてそれを継続することです。ある日、興味のあるエンジニアが過去の記事に目を通すだけでも、自社への興味度は変わります。

**オープンソースへの貢献**

OSSはエンジニア独自の文化であり、会社としてOSSに対してコミットしていることは大きなアドバンテージになり得ます。
ちょっとしたことでも良いので、コントリビュートしておくこと。そのコントリビュートをプライベートではなく会社としてサポートすること。
可能であれば、OSSのコントリビューターに寄付するなども良いでしょう。エンジニアにまつわる独自の文化を大事にする企業というアピールになります。

**エンジニアならではの環境**

大手に給料で勝つのは難しいですが、福利厚生という観点で勝負ができる場合もあります。

- 「業務の20%を自分のプロジェクトに使って良い」
- 「毎月最新の技術書が勝手に届く。書架は技術書で溢れている。」
- 「フルリモートで働けるが、いつでも出社もOK」

このような「エンジニアとして働く際には小さいけれど重要なこと」は、ことのほかエンジニアの採用にはポジティブに影響します。

## 4.3 ジョブディスクリプションの作成 - 効果的な求人票の書き方

求職者が企業に興味を惹かれる次の接点はジョブディスクリプションになります。ジョブディスクリプションの重要な点は「必要な情報を正しく書く」に尽きます。興味を惹くための言葉は、入社したあとのギャップにつながりモチベーションの低下や早期の退職につながります。なぜこのような人材を必要としているのか、入社するとどのようなパフォーマンスを期待されるか、それに見合うだけの環境や報酬はあるか、を正しく記載することが、企業と求職者の最初の信頼関係の構築につながり、採用の成功へとつながります。

例：
```markdown
# バックエンドエンジニア募集

## 企業の現状
- 創業5年のBtoB SaaS、まだ知名度ゼロ
- しかし、開発したプロダクトは月次解約率1%未満、顧客満足度は自慢です
- 技術的にはRails + Vue.js、最先端ではありません
- しかし、技術的負債と向き合う時間を確保しています

## あなたにお願いしたいこと
こんなことをやってもらいたいです
- 8年物のRailsアプリのリファクタリング（レガシーから最新へ）
- しかし、新規開発も3割程度あります
- 月1回の深夜メンテナンス当番（代休あり）
- カスタマーサポートからのバグ報告対応

## 必要なスキル
- Webアプリケーションの開発経験3年以上
- 他人のコードを読んで理解できること
- 「完璧じゃなくても、まず動かす」ができること
- チームで働くことに喜びを見いだせること

## あったら嬉しいスキル
- Rails経験（現時点で無くとも、入社後に覚えられれば問題ない）
- AWSの利用経験
- テストファーストの開発経験

## 私たちが提供できるもの
- 年収400-700万（スキルに応じて。評価は年2回）
- フルリモートOK（月1回の全体会議のみ出社いただきます）
- 書籍購入費補助（月1万円まで）
- 有給消化率85%
- エンジニア20名、約半数は子育て中
```

このような「正しくかかれた」求人票の方が、求職者には響くでしょう。

## 4.4 面接プロセスの設計 - 効果的な評価方法

### 面接プロセスの全体像 - 適切な面接回数とスピード

一昔前のIT企業は、オンライン・オフライン合わせて4〜5回、多いときは8回行っていた企業もありました。
しかし、あなたが日本の一般的な企業で働いていて中途採用を行っている場合、ほとんどの求職者は複数企業を比較しながら転職活動をしているはずです。回数もそうですが、選考に時間を掛けすぎると、優秀な候補者は途中で他社に流れてしまいます。求めている人材像と求職者がマッチしているかを十分判断できて、かつスピーディーな選考プロセスである必要があります。

**現実的なプロセス**
1. 書類選考（即日〜翌日）
   ↓ スピード勝負！選考に乗ったタイミングが求職者の熱量が一番高い状態である。

2. カジュアル面談（30分、オンライン）
   ↓ まず会社を好きになってもらうことを目的とする。選考プロセスに入るにあたり疑問が無いようにする。

3. 技術面接（60分）
   ↓ コーディングテストと、それに付随する技術的・思考プロセスを見る

4. チーム面接（60分）
   ↓ 将来の同僚と話してもらう。チームとの相性やコミュニケーション、求職者側の不安を取り除く。

5. 最終面談（30分）
   ↓ 求職者側の転職モチベーションを上げること、経営幹部の最終判断の場

6. オファー面談（30分）
   ↓ 最終条件のすり合わせの場

可能であれば、全体を2週間〜3週間以内に終わらせられるとよいです。これ以上長いと、候補者の熱意が冷めてしまい、選考中に辞退される比率があがるでしょう。。

### 書類選考・スクリーニングの効率化

書類選考は候補者の最初のスクリーニングポイントですが、見るべきポイントを押さえないと適切な判断ができず、良い候補者を見落とします。

**見るべきポイント（優先度順）：**

1. 書類の情報が適切な内容で、適切な日本語で書かれているか。
   - 不自然な日本語で書かれた書類は、ドキュメントの重要性を軽視していると考えられ、またコミュニケーションにも難があると言っているようなものです
   - 最終更新日付が古いものは、転職意欲がそもそもないか、転職に対してエネルギーを掛けられない人と判断できます。

2. 公的な情報。例えばGitHubアカウントやQiita、Zennなどの技術ブログ（あるかどうか、あれば内容をチェック）
   - 社外に情報が出ている方は、傾向として優秀な方が多いです。それが例えば、公的な企業のブログであれば、社内で一定の信頼が勝ち得ている証拠にもなります。
   - パブリックなGitHubは多くのことを教えてくれます。READMEの内容でドキュメント
   - READMEをちゃんと書いてるか
   - Issueでのコミュニケーション能力
   - 学歴なども本来はこの段階で確実に確認したいです。履歴書の記載からややはずれた経歴を歩まれている方などもいらっしゃいます。

3. これまでの学歴・社歴・転職理由の一貫性
   - 短期離職していないか、離職に際し問題行動がないかは丁寧に確認する必要はあります。
   - 「エンジニアとして成長したい」と言いながら、同じような仕事で給与が高い会社に転職している→継続してパフォーマンスを出していただけるからは給与次第とも言えます。
   - 最新技術スタックを追っている→学習意欲が高いが、何も身についていないという証明であることもあります。
   - 大手企業に所属していた、ということであれば、言える範囲で業務状況（人数、役職、社員と契約社員など）をお伺いしたいです。直近の業務でリーダー的役割を果たしていたか、どのような職種を経験されてきたかが重要です。
   - 学歴は大事ではないですが、「努力できる人かどうか」の一つの目安にはなります。
   - 大企業経験は、しっかり教育されている一方で視野が狭くなりがちです。自社の規模に合った考え方をしているかが重要です。

4. プロジェクトの具体性
   - 前項に関連していますが、直近のプロジェクトを丁寧に確認し、候補者のチーム開発における強み・弱みを確認します。
   - 業務に関する具体的な数字がどこまで頭の中に入っているかを確認できるとよいでしょう。

5. 資格、スキル
   - 資格で能力を測ることはできませんが、一つの目安になります。「勉強を続けられるか」「継続的にスキルを習得しようとしているか」を推し量ることが出来ます。
   - 書類に記載されているスキルについても同様ですが、本当に業務に必要な技術スキルは面談・面接でヒアリングすることで確認しましょう。

### 技術評価の方法

技術評価を短時間で行うことは難しいですが、最低限の確認をするためのいくつかの方法はあります。コーディングテストや面接での質問は、チームにジョインする上で絶対に外せないポイントをあらかじめ絞った上で、候補者に確認すると良いです。

**1. コーディングテスト**

面接の前に自宅で解いてもらうコーディングテストは、候補者の負担が少ないという利点がありますが、それでも時間を掛けないと解けないような問題にするべきではありません。
また、直近はAIの活用によりコーディングテストの存在意義が問われていますが、AIの利用を制限することは事実上難しいので、AI活用を許容した上で、どのような思考を経て解に至ったのかを面接で確認するとよいでしょう。

```
設計原則：
- 可能であれば実務に即した問題であること
- 2-4時間で解ける範囲であること（長いと求職者は離脱します）
- 複数の解法が存在すること（応用力が見られます）
- 部分点の設定（「考え方としてどの部分が網羅されていればよいか」を事前に決めておくと、判断する時間が短縮できます）

評価基準：
- 設計（20%）
- 要件に合った機能の実装（30%）
- 可読性と保守性（30%）
- パフォーマンス（10%）
- テストコード（10%）
```

**2. ライブコーディング**

ライブコーディングは、面接官と共同してプログラミングをその場で行うタイプの技術評価手法です。ここで確認するのは、ソフトウェアを正しく動作させるためにどのように考えるかの思考プロセスになります。
このとき「分からない」と正直に言えるか、どの部分が分からないと説明できるか、ヒントを受けて軌道修正できるか、問題を分解して考えられるかを確認することで、実際にチームで働く際のスキルを見極めることができます。

```
実施方法：
- ホワイトボードや画面共有でのペアプログラミングの形式で行う
- 簡単な問題から開始する（ライブコーディングは得てして緊張します）
- 思考プロセスを観察する
- ときにはヒントの提供も行う（詰まっていても沈黙で追い込まない）

観察ポイント：
- 問題理解力
- アプローチの選択
- デバッグ能力
- コミュニケーション
```

**3. システム設計面接**

特にシニアレベル以上の候補者に向けに行う、コーディングを伴わない思考プロセスの確認をする手法です。観点はライブコーディングと同様ですが、より正解がない質問になりますので、プロセスを評価することになります。
システム設計面接と銘打っていますが、実際は要件定義からソフトウェア開発の上流工程をなぞりコーディングの手前までを確認します。

```
問題例：
「Facebookのようなサービスを設計するとき、何を行いますか？」

評価項目：
- 要件定義の明確化
- スケーラビリティの考慮（要件に合わせたシステム選定ができるか）
- トレードオフの理解（メリットとリスクの判断）
- 技術選定の根拠（複数の技術から要件に適した技術を選択できているか）
```

### 面接

面接は双方のインタラクティブな情報交換になります。候補者が持つ能力を正確に把握するためにも、短時間で適切な問いを行う必要があります。
「話しやすい」安心感を与え、こちらから会社の状況や求めるスキルを提示しつつ、候補者のスキルや経験を引き出せるようなインタビューにしましょう。

#### いままでの行動におけるインタビュー（Behavioral Interview）

ここで役立つのがSTARメソッドです。

**STARメソッド：**
質問に対して、論理的で説得力のある説明ができているかどうかのフレームワークになります。
候補者の説明で足りない部分があれば、このフレームワークに沿って質問を重ねると良いでしょう。

- **S**ituation（状況）: どのような背景や状況だったか（それらを簡潔に説明できているか）
- **T**ask（課題）: 直面した課題、または達成すべき目標は何だったか
- **A**ction（行動）: 課題解決のために「候補者自身」が取った具体的な行動は何か
- **R**esult（結果）: 行動によって得られた定量的・定性的な成果や学びは何か

**質問例：**
```
リーダーシップ：
「技術的に困難な決定を下した経験を教えてください」

協働：
「メンバー同士の意見の対立や認識の齟齬があった経験があれば、その経験と対処法を教えて下さい」

問題解決：
「本番環境での重大な障害対応の経験はありますか？」

学習能力：
「新しい技術を短期間で習得した経験は？」
```

### カルチャーフィット評価

カルチャーフィットは一概に説明をすることはできませんが、現在のエンジニアチームに参画した場合に活躍できるかは重要な確認点となります。
一方で、現在のカルチャーには合わなくとも「チームを変革できる」ことが求められる（そのようなメンバーを求めている）のであれば、完全なカルチャーフィットは必要ではないでしょう。
自社のMission, Vision, Valueや、社風、チーム状況を加味しながら、候補者の性格や志向を見極めます。

**評価の観点：**
```
価値観が一致しているか：
- チームワーク vs 個人主義のバランス
- 品質 vs スピードのバランス
- イノベーション vs 安定性のバランス

働き方：
- 自律性のレベル
- フィードバックへの受容性
- 学習意欲
- 思考の柔軟性
```

## 4.5 面接官のトレーニング - 面接の質をチームで上げる

### 面接スキルの標準化

「技術力が高い人＝良い面接官」では当然ありません。候補者の良いところを引き出し、見極め、入社してから活躍できることを根拠を持って説明できるのが良い面接官です。

候補者体験という考え方があります。
面接は候補者にとっては「会社とのインタフェース部分」になります。適切な説明をする一方で、候補者の心理的な負担を減らし、的確な質問をして、公平に評価するスキルが必要になります。
注意すべきは、これらは訓練で身につくスキルであるということ、そして多くのエンジニアはこのトレーニングを苦手としているところです。

**面接官トレーニングプログラム：**

1. 面接基礎研修（1時間）

最初に、なぜ採用をするのか、採用における面接の目的と重要性について説明します。「構造化面接の重要性」「無意識のバイアス」「法的に聞いてはいけない質問」など、知識としてインプットが必要な内容について理解してもらいます。

2. 面接への同席

すでに面接官として経験のあるエンジニアの横で実際の面接を観察します。徐々に一部の質問を担当するなど、段階的に参加度を上げます。可能であれば、終了後にフィードバックを行い「あの場面ではこう聞いたほうがよかった」といった具体的なアドバイスをします。

3. 実践とレビュー

独り立ちした後も、定期的に面接の質をレビューします。可能であれば他の面接官と同席して、終了後にお互いの質問の仕方や評価の妥当性について議論する機会を設けます。


### 公平な評価 - バイアスを取り除く

人間が人間を評価する以上、その評価にはどうしてもバイアスはかかります。その上でどのようなバイアスがあるかを理解した上で、面接の際には「バイアスがかかるものだ」と念頭においたうえで評価に影響しないよう注意しましょう。

**一般的なバイアスと対策：**

| バイアス | どういうこと | どう防ぐか |
|---------|-------------|-----------|
| 確証バイアス | 第一印象が良いと、その後も良い面を重視してしまう | 全候補者に同じ質問をする構造化面接を行う |
| ハロー効果 | 出身大学や経験している企業だけで全体を高く評価する | 複数の面接官で独立して評価する |
| 類似性バイアス | 自分と似た経歴・考え方の人を高く評価する | 違うバックグラウンドを持つ面接官をアサインする |
| アンカリング | 例：年収希望額に引きずられた評価をする | 評価に関連する情報（例は報酬）との議論を分離する |


### その他、面接にて気をつけておくべきこと

前章までに説明した内容以外に、面接を円滑に進め、適切な評価をするために気をつけるべきポイントをまとめます。

- メモを取る
- 候補者の名前を覚えて呼びかける
- 候補者をリラックスさせるような会話、話し方
- 質問は準備し、明確に質問する
- 十分な時間を与える（沈黙も許容し、必要があれば促すようなヒントを出す）
- 候補者からの質問に丁寧に答える
- 違法となる質問・個人的な質問をしない（年齢、家族構成等）
- 圧迫的な態度を取らない
- 他社の悪口を言わない
- 約束できないことを言わない

## 4.6 採用の意思決定プロセス

### 評価の集約

面接が終わった後の意思決定プロセスは、採用の質、ひいては組織の質を左右する重要なステップです。ここをなんとなくで進めてしまうと、偏った意思決定となり適切な判断が下せなくなります。

**採用評価チーム：**

採用を進める際に、候補者を適切に評価するためのチームを構成します。

メンバー：
- 人事マネージャー（最終決定者。EMやそれ以上の役職が多い）
- 技術面接担当者
- カルチャー面接担当者
- 人事担当者
- 面接や面談に参加したチームメンバー（将来一緒に働く同僚としての評価）

これらのメンバーで、面接後にミーティングにて評価します。重要なのは、ミーティング前に一度各面接官が独立して事前評価することです。他の面接官の意見を聞いてしまうと、アンカリングが起きます。

### 事前評価の5段階

事前評価は5段階で行うと、面接後のミーティングのすり合わせや、その後の評価軸の修正に役立ちます。

Strong Yes - 絶対に採用すべき。このレベルの人は年に数人しか出会わない
Yes - 採用を推奨する。チームに貢献できる
Neutral - 判断がつかない。追加情報が必要
No - 採用すべきでない。具体的な懸念がある
Strong No - 絶対に採用すべきでない。明確なリスクがある


### 面接後のミーティング

面接後のミーティングでは、単なる「良い・悪い」の報告ではなく、候補者を多角的に見て全体像を把握し、意思決定をする場になります。

進行は以下のように行うとよいでしょう。

1. 各面接官の評価共有（理由含む）
2. 相違点の議
3. 追加情報の必要性検討（もう一段階面接をするか？確認事項はあるか？）
4. リスクとポテンシャルの評価
5. 最終決定

ここでは議論のルールとして「データと事実に基づき、具体例を挙げ」て議論するということを念頭に起きます。たとえば「なんとなく合わない気がする」という場合も、より言語化して「面接中に〜という発言があり、チームの価値観と合わない」といった、具体的な観察に基づいた説明をするようにします。

また、一つ大事なルールとして、 **事前評価でStrong Noが一人でもいたら、その理由を徹底的に議論する** というものがあります。
Strong Noは絶対拒否という意味合いで、この人とは働けないという意思表明になります。技術力だけでなく、カルチャーフィットや相性なども考慮した上で、適切な評価を行います。

### リファレンスチェック

上位ポジションへオファーする場合は、リファレンスチェックを行うことをお勧めします。役職が上であればあるほど、採用の失敗によるリスクは高まります。候補者の了承を得た上で、以前の同僚や上司の所感を確認します。

** リファレンスチェックでの質問例：**

関係性の確認：
「どのような関係で、どれくらい一緒に働きましたか？」

強みの確認：
「彼/彼女の最も優れた点を3つ挙げてください」

改善点の確認：
「成長の余地がある領域はどこですか？」

再雇用の意思：
「機会があれば、また一緒に働きたいですか？」

具体例：
「印象的なプロジェクトや成果を教えてください」

## 4.7 オファーと交渉

**競争力のあるオファーを出す**

良い候補者が見つかり、無事最終面接を通過しました。しかし、当然オファーが魅力的でなければ辞退されます。

オファーパッケージは給与だけではありません。候補者が何を重視するかは人によって異なります。

**年収：**

市場レートの調査は必須です。転職市場の相場を知らずにオファーを出すと、候補者に「この会社は市場を理解していない」と思われます。一方で、社内の公平性も一定考慮する必要はあります。案外、社員同士はお互いの年収を話したりします。

** 給与以外のポイント： **

交渉の前に、事前に「給与よりもこちらの方が大事」というポイントを押さえておきます。

- 入社時期の柔軟性（「来月からではなく、2ヶ月後にしてほしい」）
- リモートワークの割合（「週3日在宅なら年収が少し下がっても構わない」）
- 学習予算（「カンファレンス参加費を出してくれるなら」）
- 職位/タイトル（「シニアというタイトルをもらえるなら」）

これらの調整は、給与の調整よりも比較的簡単です。チームに必要な人材であることを強くアピールして、社内で交渉してみるべきです。

### 交渉の戦略

交渉を「勝ち負け」ではなく「すり合わせ」です。候補者はそれこそ人生を掛けて交渉しますので、入社前に遺恨を残さないよう、双方が納得し満足する条件で入社していただけるようにしましょう。

交渉のポイント：

透明性 - 給与レンジがあるなら正直に伝える
柔軟性 - 希望の給与が出せないなら、他の条件で補えないか考える
事情の理解 - 「なぜその金額を希望するのか」を確認することで本当のニーズが異なる場合がある
長期視点 - 「入社時の条件」だけでなく「半年後の評価で見直す」という選択肢も示す

### オファーレターに同僚からのメッセージ

実際のところ、候補者が複数の企業から内定を獲得しておりオファーの条件的にもほぼ同じ、となった場合、最後は感情的での判断になります。一緒に働いて楽しそうか、気持ちよく働けるかをアピールする最後の機会として、オファーレターに同僚からのメッセージを入れましょう。（これを怠って、本当に良い候補者の内定承諾を獲得できず他社に流れてしまった経験があります）


## 4.8 オンボーディング

採用は入社すれば終わりではなく、チームで活躍するまでが採用だともいえます。実際に、採用エージェントの手数料支払は「入社して半年間退職しない」ことが条件になっていることも多く、最終的にチームに定着しなければ採用は成功したとはいえません。
チームへの定着のためにオンボーディングの仕組みは重要です。新しいメンバーが入社し、初日にPCを渡して「このWikiを読んでおいて」と伝えただけでは、数日後には「何をすればいいか分からない」と途方に暮れてしまいます。

最低でも30日、60日、90日のオンボーディングプランを作り、メンターを設定して伴走する体制を整えましょう。

**最初の30日：「安心して慣れる」**

この時期に大切なのは、技術的なキャッチアップよりも「ここにいていいんだ」という安心感になります。

- 環境セットアップを一人で苦労させない（手順書を用意し、メンターがサポート）
- チーム全員との1on1を2週間以内に実施（顔と名前を一致させる）
- プロダクトの全体像を説明する（コードを書く前に「何のために」を理解する）
- 小さなタスクを割り振る（最初の成功体験が自信になる）

**60日まで：「貢献し始める」**

少しずつ「自分の仕事」が増えてくる時期です。
- 最初の機能開発に取り組む（メンターのサポート）
- コードレビューに参加する（レビューする側にも回ってもらう）
- ドキュメントの改善を依頼する（新鮮な目で見たほうが改善点が見つかる）
- 週次の1on1でフィードバックを密に行う

**90日まで：「チームの一員になる」**

独り立ちの時期です。
- 独立したプロジェクトを任せる
- プロセス改善の提案を歓迎する（「新しい人の視点」は組織にとって貴重）
- チームへの貢献を認め、フィードバックする
- 90日レビューを実施し、今後のキャリアプランを話し合う

### メンター - 「聞ける人」がいるだけで安心する

新しくジョインしたメンバーにとって一番辛いのは「どこまで聞いていいのか分からない」と思ってしまうことだと思います。メンター制度はそのような心理的ハードルを下げ、円滑なチームへの参画を導くためにあります。

メンターの役割は、技術的な側面のサポートだけではありません。むしろ、ちょっとした情報や暗黙知的な「会社のルール」を教えることにあります。チームの小さな情報を知っているか知らないかで、新メンバーの居心地を大きく左右します。

*** メンターの候補者： ***

- 1年以上在籍していて、会社の文化を理解している
- コミュニケーション力が高い（技術力より重要）
- 時間的に余裕がある（忙しすぎる人は避ける）
- 新メンバーと近いレベルだと相談しやすい（シニアすぎると敷居が高い）

メンター期間は3ヶ月を目安にしていますが、その後も自然と関係が続くケースが多いです。また、メンターの経験はメンター側にとっても成長の機会ででもあります。

## 4.8 パフォーマンス評価制度

### 評価制度の設計原則

**効果的な評価制度の特徴：**
- 公正性と透明性
- 予測可能性
- 成長志向
- 定量と定性のバランス
- 継続的フィードバック

### 評価フレームワーク

**多面的評価の構成：**
```
評価の構成要素：
1. 成果（What）- 40%
   - OKR達成度
   - プロジェクト成果
   - 品質指標

2. 行動（How）- 30%
   - コラボレーション
   - リーダーシップ
   - イノベーション

3. スキル成長 - 20%
   - 技術スキル
   - ソフトスキル
   - 新領域への挑戦

4. 組織貢献 - 10%
   - メンタリング
   - 知識共有
   - 採用活動
```

### OKRとKPIの活用

**OKR設定の例：**
```
Objective: 
システムの信頼性を大幅に向上させる

Key Results:
- KR1: システム稼働率を99.9%から99.99%に向上
- KR2: 平均復旧時間（MTTR）を30分から10分に短縮
- KR3: 自動化されたテストカバレッジを70%から90%に向上
```

**個人KPIの例：**
```
エンジニアレベル別KPI：

ジュニア：
- コードレビューの完了時間
- バグ修正率
- 学習目標の達成

ミドル：
- 機能デリバリー数
- コード品質スコア
- ペアプログラミング時間

シニア：
- アーキテクチャ提案数
- メンタリング時間
- 技術的負債の削減
```

### 360度フィードバック

**フィードバック源：**
- 上司（40%）
- 同僚（30%）
- 部下（20%）
- 自己評価（10%）

**質問例：**
```
評価項目（1-5スケール）：
- 技術的専門性を発揮している
- チームワークを大切にしている
- 締切を守る
- 建設的なフィードバックを提供する
- 継続的に学習している

自由記述：
- 最も優れている点は？
- 改善すべき点は？
- 今後期待することは？
```

## 4.9 評価の実施とキャリブレーション

### 評価サイクル

**年間スケジュール例：**
```
Q1（1-3月）：
- 1月：目標設定
- 2月：進捗確認
- 3月：Q1レビュー

Q2（4-6月）：
- 4月：中間フィードバック
```

## まとめ

採用と評価は、エンジニアリングマネージャーの最も重要な責任の一つです。優秀な人材を獲得し、公正に評価し、継続的に成長させることで、強いエンジニアリングチームを構築できます。

採用においては、単にスキルだけでなく、文化適合性や成長可能性を評価することが重要です。また、採用プロセス自体が候補者体験となり、企業ブランドに影響することを忘れてはいけません。

評価においては、透明性と公正性が鍵となります。明確な基準、継続的なフィードバック、成長機会の提供により、メンバーのモチベーションと成長を促進できます。

最後に、採用と評価は継続的な改善が必要なプロセスです。データを収集し、フィードバックを受け、常により良い方法を模索し続けることが、長期的な成功につながります。

次章では、プロジェクトマネジメントについて詳しく見ていきます。優秀な人材を活かし、効果的にプロジェクトを遂行することで、ビジネス価値を創出します。
