# 第4章：採用と評価 - 優秀な人材の獲得と評価制度

## はじめに

「採用は、マネージャーが行う仕事で最も重要なものの一つである」
これは私の上司や同僚のマネージャーから、事ある毎に交わされた言葉です。

一人の優秀なエンジニアの採用は、開発チームの生産性を飛躍的に向上させますが、逆に不適切なメンバーの採用は、チーム全体のパフォーマンスを著しく低下させ、その修復には膨大なコストと時間がかかります。また、評価についても同じ事が言えます。公正で透明な評価制度は、優秀な人材を維持し、成長を促進する鍵となります。一方で、評価の仕組みが機能していないと、メンバーの成長機会を奪い、信頼関係を損なうことになります。

本章では、採用から評価まで、人材マネジメントの核心部分について考えていきます。テンプレートやフレームワークも紹介いたしますが、それらを機械的に当てはめることはせず、「このチームで働きたいと思ってもらえるか」「このメンバーが成長するために何ができるか」という問いに真摯に向き合うことが大切であると考えます。

## 4.1 採用の重要性と影響

### 採用の乗数効果

スティーブ・ジョブズは「A級の人材はA級の人材を呼ぶが、B級の人材はC級の人材を採用する」と言いました。この言葉はやや極端ではありますが、採用の連鎖的な影響を端的に表しているとも言えます。

技術力は申し分無いのに、コミュニケーションに難があるエンジニアを採用した、ということを考えてみましょう。会社のフェーズによっては、またはチームの状況によっては機能することもあるかもしれませんが、多くの場合は失敗する未来が見えるのではないでしょうか。攻撃的な会話はメンバーのコミュニケーションを萎縮させ、チーム全体の生産性を落とす結果にもなります。

逆に、技術力はまあまあでも学習意欲が高くチームを盛り上げてくれる人を採用した場合。チーム内での知識共有が活発になり、結果として全員のスキルが上がる、なんてことも大いに想像できるかと思います。

**優秀な採用がもたらす効果：**
- チーム全体のレベル向上（優秀なエンジニアは周囲の生産性も引き上げる）
- イノベーションの促進（新しい視点が新しいアイデアを生む）
- 学習文化の強化（あの人のようになりたい、という良い競争心）
- 採用ブランドの向上（著名で実力のあるエンジニアの存在）
- ネットワーク効果（優秀な人は優秀な知り合いを紹介してくれる）

**不適切な採用のコスト：**
- 生産性の低下（チーム全体の25-30%とも言われています）
- モラルへの悪影響（「なぜあの人が採用されたのか」という不信感）
- マネジメント時間の増大（本来必要のないマネジメントによって他の業務時間を奪われます）
- 再採用コスト（採用人件費などを考慮すると年収の1.5-2倍は損失になります）
- チーム文化の劣化（一度壊れた文化の修復は本当に大変です）と

### 採用におけるEMの役割

EMの仕事は多岐にわたりますが、この章の最初で紹介したように、採用はEMが行う仕事で最も重要なものの一つです。今日採用した人は3年後もそのチームにいるとしたら。その人の存在が文化を形作り、プロダクトの品質を左右し続けるのです。EMはその採用において責務を追うことになります。

EMが採用においてどのように関わるか、については、大きく3つのレベルに分けられます。

**戦略レベル：**
どのような開発をするのか、そのためにどのようなチームを作りたいのか。
- 人員計画の策定（半年〜数年）
- 採用基準の定義（どのような人をいま求めるのか）
- 採用ブランディング（中長期的にどのようなチームを作るのか・そのために何をブランディングするか）
- 多様性戦略（広い母集団から変化に強いチームを作る）

**戦術レベル：**
採用プロセスそのものの設計。
- ジョブディスクリプション作成（具体的にどのような仕事を任せたいか）
- 面接プロセス設計（人材を正しく見極めるための仕組み）
- 面接官のトレーニング（適切な判断ができるようにするため）
- 最終意思決定（最低限必要な要素、採用できる幅、許容できる幅）

**実行レベル：**
実際に採用を進めるプロセスの実施。
- スクリーニング（多くの候補者から条件に足る人を絞込む）
- 技術面接の実施（必要な技術力があるとみなせるだけの技術面接）
- 候補者体験の管理（親しみと信頼の両軸を採用まで一貫して）
- オファー交渉（年収テーブルと見立てたスキルの調整・競合他社との奪い合い）

## 4.2 人員計画と採用戦略

### 人員計画の策定
「人が足りません」と言うのは簡単ですが、「どのような人が何人足りないのか」「なぜ足りないのか」「いつまで必要なのか」を的確に説明できないと、採用活動に説得力が出ません。

前章では採用の関わり方について3つのレベルに分類しましたが、まずは計画を立てる必要があります。以下のステップで考えると良いでしょう。

計画フレームワーク：
1. ビジネス目標の理解
まず、この先半年〜1年で何を達成しなければならないのかを明確にします。新機能の開発、技術的負債の解消、スケーラビリティ対応など、具体的な目標を洗い出します。

2. 必要スキルの特定
目標を達成するために、どんなスキルセットが必要かを考えます。技術スタック、ドメイン知識、経験レベルの分布。「フロントエンドエンジニアが2人必要」ではなく、「Reactの経験があり、アクセシビリティの知識があるミドルレベルのエンジニアが2人必要」といった、具体的なスキルが判定できる解像度で設定します。

3. タイムラインの設定
採用には時間がかかります。求人を出してから入社までの平均リードタイムは3〜6ヶ月。ここにオンボーディング期間（通常1〜3ヶ月）を加えると、「来月からフル稼働してほしい」は現実的な計画ではありません。プロジェクトのマイルストーンから逆算して、いつまでに採用を完了すべきかを計画します。

4. 予算の確保
給与レンジ、採用コスト（エージェントへの手数料、求人広告費など）、採用してから実務に入るまでのトレーニング費用など、これらを具体的な数字にします。この具体的な数値から「何人必要か」の議論が「いくら必要か」という具体的な話に変わります。

### 採用チャネルの最適化 - 効果的な採用経路の選択

エンジニアを採用するにあたっては複数の経路があります。これを採用チャネルと呼びます。2026年現在での各チャネルの実態は以下のようになります。

**リファラル（社員紹介）**
「優秀な人は優秀な人を知っている」というのは前段でもご紹介したとおりで、獲得した人材が優秀であることを求めるのであればこれがもっとも確実です。しかし、リファラルを進めるには条件が揃っている必要はあります。普段から「今このような人材を探しています」という情報を共有することが重要です。また、メンバーが「自分の会社になら知り合いを入れてもいい」と思えるような環境である必要があります。ただ「紹介してくれたら3万円」といった制度だけ作っても全く機能しません。

**直接スカウト**
GitHubのコードやSNSの発信を見て「この人は素晴らしい」という人にDMを送ります。返信率は10%を切りますが、タイミングが合えば（その方がたまたま転職を検討しているなどしていれば）非常に優秀な人材を確保することができます。その際、メッセージはその人のコードの何が素晴らしいか、具体的に書いて興味を惹かれるものでないといけません。定型文は絶対に✕です。

**採用エージェント**
採用人件費の外注化、という観点では必要なチャネルではありますが、費用対効果という点では難しいものがあります。年収の35%程度、採用難易度が高い場合は50%以上の手数料がかかります。半年以内の退職は手数料の支払いがない、という契約が多いですが、すぐに退職されると大きな損失になります。しかし「3ヶ月後までに3人採用しないとプロジェクトが回らない」といった緊急時には、採用のテーブルに乗る母数が増える、という観点では頼りがいがあります。ここではエンジニア採用に強い専門エージェントを選ぶことが活用の鍵になります。

**求人サイト・ダイレクトリクルーティング**
WantedlyやGreen、Bizreachなどは、転職したい本人が直接企業と接点を持てる仕組みとなっており、直接スカウトなどに比べると求職者とのコンタクトを多く取れる利点があります。一方で、応募はあってもスクリーニングが大変です。100人応募があっても、面接したいと思える人は5人程度です。しかし、その5人の中から1人採用できれば成功と言えるでしょう。

**テックイベント・勉強会**
採用広報という考え方では、イベントや勉強会を使うのは長期的な効果はあります。メンバーが登壇したり、会社がスポンサーしたりすることで認知度を上げ、「あの会社の○○さんは技術力が高い」と思ってもらえれば、リファラル採用など他の採用チャネルへの良い影響をだすことができます。可能であればイベントの場で連絡先を交換し、定期的に連絡できる仕組みを作るとよいでしょう。ただし、イベントの際は採用目的を前面に出さず、純粋に技術交流することが重要です。

### 採用ブランディング - 小さい企業でも知名度を積み重ねる

GoogleやMercariのようなブランド力があれば採用もスムーズに行きますが、小さな会社だからこそできるブランディングもあります。

**技術ブログの定期的な執筆**

エンジニアは最新の技術動向を常に追いかけますが、必ずしも最新の情報である必要はありません。現場に近いリアルな話や、思いもつかないような失敗談、あるいはオンボーディングで淡々と進めたこと、などでも構いません。重要なのは「エンジニアが興味を引くトピックである」こと、そしてそれを継続することです。ある日、興味のあるエンジニアが過去の記事に目を通すだけでも、自社への興味度は変わります。

**オープンソースへの貢献**

OSSはエンジニア独自の文化であり、会社としてOSSに対してコミットしていることは大きなアドバンテージになり得ます。
ちょっとしたことでも良いので、コントリビュートしておくこと。そのコントリビュートをプライベートではなく会社としてサポートすること。
可能であれば、OSSのコントリビューターに寄付するなども良いでしょう。エンジニアにまつわる独自の文化を大事にする企業というアピールになります。

**エンジニアならではの環境**

大手に給料で勝つのは難しいですが、福利厚生という観点で勝負ができる場合もあります。

- 「業務の20%を自分のプロジェクトに使って良い」
- 「毎月最新の技術書が勝手に届く。書架は技術書で溢れている。」
- 「フルリモートで働けるが、いつでも出社もOK」

このような「エンジニアとして働く際には小さいけれど重要なこと」は、ことのほかエンジニアの採用にはポジティブに影響します。

## 4.3 ジョブディスクリプションの作成 - 効果的な求人票の書き方

求職者が企業に興味を惹かれる次の接点はジョブディスクリプションになります。ジョブディスクリプションの重要な点は「必要な情報を正しく書く」に尽きます。興味を惹くための言葉は、入社したあとのギャップにつながりモチベーションの低下や早期の退職につながります。なぜこのような人材を必要としているのか、入社するとどのようなパフォーマンスを期待されるか、それに見合うだけの環境や報酬はあるか、を正しく記載することが、企業と求職者の最初の信頼関係の構築につながり、採用の成功へとつながります。

例：
```markdown
# バックエンドエンジニア募集

## 企業の現状
- 創業5年のBtoB SaaS、まだ知名度ゼロ
- しかし、開発したプロダクトは月次解約率1%未満、顧客満足度は自慢です
- 技術的にはRails + Vue.js、最先端ではありません
- しかし、技術的負債と向き合う時間を確保しています

## あなたにお願いしたいこと
こんなことをやってもらいたいです
- 8年物のRailsアプリのリファクタリング（レガシーから最新へ）
- しかし、新規開発も3割程度あります
- 月1回の深夜メンテナンス当番（代休あり）
- カスタマーサポートからのバグ報告対応

## 必要なスキル
- Webアプリケーションの開発経験3年以上
- 他人のコードを読んで理解できること
- 「完璧じゃなくても、まず動かす」ができること
- チームで働くことに喜びを見いだせること

## あったら嬉しいスキル
- Rails経験（現時点で無くとも、入社後に覚えられれば問題ない）
- AWSの利用経験
- テストファーストの開発経験

## 私たちが提供できるもの
- 年収400-700万（スキルに応じて。評価は年2回）
- フルリモートOK（月1回の全体会議のみ出社いただきます）
- 書籍購入費補助（月1万円まで）
- 有給消化率85%
- エンジニア20名、約半数は子育て中
```

このような「正しくかかれた」求人票の方が、求職者には響きます。

## 4.4 面接プロセスの設計 - 効果的な評価方法

### 面接プロセスの全体像 - 適切な面接回数とスピード

一昔前のIT企業は、オンライン・オフライン合わせて4〜5回、多いときは8回行っていた企業もありました。
しかし、あなたが日本の一般的な企業で働いていて中途採用を行っている場合、ほとんどの求職者は複数企業を比較しながら転職活動をしているはずです。回数もそうですが、選考に時間を掛けすぎると、優秀な候補者は途中で他社に流れてしまいます。求めている人材像と求職者がマッチしているかを十分判断できて、かつスピーディーな選考プロセスである必要があります。

**現実的なプロセス**
```
1. 書類選考（即日〜翌日）
   ↓ スピード勝負！選考に乗ったタイミングが求職者の熱量が一番高い状態である。
2. カジュアル面談（30分、オンライン）
   ↓ まず会社を好きになってもらうことを目的とする。選考プロセスに入るにあたり疑問が無いようにする。
3. 技術面接（60分）
   ↓ コーディングテストと、それに付随する技術的・思考プロセスを見る
4. チーム面接（60分）
   ↓ 将来の同僚と話してもらう。チームとの相性やコミュニケーション、求職者側の不安を取り除く。
5. 最終面談（30分）
   ↓ 求職者側の転職モチベーションを上げること、経営幹部の最終判断の場
6. オファー面談（30分）
   ↓ 最終条件のすり合わせの場
```

可能であれば、全体を2週間〜3週間以内に終わらせられるとよいです。これ以上長いと、候補者の熱意が冷めてしまいます。

### 書類選考・スクリーニングの効率化
書類選考
100通の応募が来たとき、どのように選ぶべきでしょうか。

**見るべきポイント（優先順）：**

1. **GitHubアカウント**（あれば最優先でチェック）
   - コードの質 > コミット数
   - READMEをちゃんと書いてるか
   - Issueでのコミュニケーション能力

2. **転職理由の一貫性**
   - 「成長したい」が3社連続 → 何か問題あるかも
   - 技術スタックを追っている → 学習意欲が高い
   - 全部1年で辞めてる → ちょっと心配

3. **プロジェクトの具体性**
   - 「大規模システム開発」→ 具体的に何？
   - 「性能を改善」→ 何を何％改善？
   - 数字で語れる人は信頼できる

**意外と重要じゃないこと：**
- 学歴（東大卒でもコードが書けない人はいます）
- 大企業経験（むしろ手を動かしていない可能性があります）
- 資格（基本情報技術者試験など、正直あまり重要ではありません）

### 技術評価の方法

**1. コーディングテスト**
```
設計原則：
- 実務に即した問題
- 2-4時間で解ける範囲
- 複数の解法が存在
- 部分点の設定

評価基準：
- コードの正確性（40%）
- 可読性と保守性（30%）
- パフォーマンス（20%）
- テストコード（10%）
```

**2. ライブコーディング**
```
実施方法：
- 画面共有でのペアプログラミング
- 簡単な問題から開始
- 思考プロセスの観察
- ヒントの提供

観察ポイント：
- 問題理解力
- アプローチの選択
- デバッグ能力
- コミュニケーション
```

**3. システム設計面接**
```
問題例：
「Twitterのようなサービスを設計してください」

評価項目：
- 要件定義の明確化
- スケーラビリティ考慮
- トレードオフの理解
- 技術選定の根拠
```

### 行動面接（Behavioral Interview）

**STARメソッド：**
- **S**ituation：状況の説明
- **T**ask：課題や目標
- **A**ction：取った行動
- **R**esult：結果と学び

**質問例：**
```
リーダーシップ：
「技術的に困難な決定を下した経験を教えてください」

協働：
「意見の対立をどのように解決しましたか？」

問題解決：
「本番環境での重大な障害対応の経験は？」

学習能力：
「新しい技術を短期間で習得した経験は？」
```

### カルチャーフィット評価

**評価の観点：**
```
価値観の一致：
- チームワーク vs 個人主義
- 品質 vs スピード
- イノベーション vs 安定性

働き方：
- 自律性のレベル
- フィードバックへの receptiveness
- 学習意欲
- 柔軟性
```

## 4.5 面接官のトレーニング

### 面接スキルの標準化

**面接官トレーニングプログラム：**
```
1. 基礎研修（2時間）
   - 面接の目的と重要性
   - 法的注意事項
   - バイアスの認識

2. シャドーイング（3-5回）
   - 経験豊富な面接官の観察
   - 段階的な参加
   - フィードバック

3. 実践とレビュー
   - 録画レビュー
   - ペアでの実施
   - 継続的改善
```

### バイアスの軽減

**一般的なバイアスと対策：**

| バイアス | 説明 | 対策 |
|---------|------|------|
| 確証バイアス | 第一印象を裏付ける情報を探す | 構造化面接 |
| ハロー効果 | 一つの良い点が全体評価に影響 | 複数評価者 |
| 類似性バイアス | 自分に似た人を高評価 | 多様な面接官 |
| アンカリング | 最初の情報に引きずられる | 独立評価 |

### 面接の実施ガイドライン

**Do's：**
- 候補者をリラックスさせる
- 明確な質問をする
- 十分な時間を与える
- メモを取る
- 一貫した評価基準を使う

**Don'ts：**
- 個人的な質問をしない
- 違法な質問を避ける（年齢、家族構成等）
- 圧迫的な態度を取らない
- 他社の悪口を言わない
- 約束できないことを言わない

## 4.6 意思決定プロセス

### 評価の集約

**採用委員会の構成：**
```
メンバー：
- Hiring Manager（決定権）
- 技術面接官（2-3名）
- カルチャー面接官
- HR担当者
- （可能なら）チームメンバー

評価フォーマット：
各面接官が独立して評価
- Strong Yes：絶対採用すべき
- Yes：採用推奨
- Neutral：どちらでもない
- No：採用しない方が良い
- Strong No：絶対採用すべきでない
```

### デブリーフィング

**効果的なデブリーフィング：**
```
進行フロー：
1. 各面接官の評価共有（理由含む）
2. 相違点の議論
3. 追加情報の必要性検討
4. リスクとポテンシャルの評価
5. 最終決定

議論のルール：
- データと事実に基づく
- 具体例を挙げる
- 感情的な判断を避ける
- 全員の意見を聞く
```

### リファレンスチェック

**効果的な質問：**
```
関係性の確認：
「どのような関係で、どれくらい一緒に働きましたか？」

強みの確認：
「彼/彼女の最も優れた点を3つ挙げてください」

改善点の確認：
「成長の余地がある領域はどこですか？」

再雇用の意思：
「機会があれば、また一緒に働きたいですか？」

具体例：
「印象的なプロジェクトや成果を教えてください」
```

## 4.8 パフォーマンス評価制度

### 評価制度の設計原則

**効果的な評価制度の特徴：**
- 公正性と透明性
- 予測可能性
- 成長志向
- 定量と定性のバランス
- 継続的フィードバック

### 評価フレームワーク

**多面的評価の構成：**
```
評価の構成要素：
1. 成果（What）- 40%
   - OKR達成度
   - プロジェクト成果
   - 品質指標

2. 行動（How）- 30%
   - コラボレーション
   - リーダーシップ
   - イノベーション

3. スキル成長 - 20%
   - 技術スキル
   - ソフトスキル
   - 新領域への挑戦

4. 組織貢献 - 10%
   - メンタリング
   - 知識共有
   - 採用活動
```

### OKRとKPIの活用

**OKR設定の例：**
```
Objective: 
システムの信頼性を大幅に向上させる

Key Results:
- KR1: システム稼働率を99.9%から99.99%に向上
- KR2: 平均復旧時間（MTTR）を30分から10分に短縮
- KR3: 自動化されたテストカバレッジを70%から90%に向上
```

**個人KPIの例：**
```
エンジニアレベル別KPI：

ジュニア：
- コードレビューの完了時間
- バグ修正率
- 学習目標の達成

ミドル：
- 機能デリバリー数
- コード品質スコア
- ペアプログラミング時間

シニア：
- アーキテクチャ提案数
- メンタリング時間
- 技術的負債の削減
```

### 360度フィードバック

**フィードバック源：**
- 上司（40%）
- 同僚（30%）
- 部下（20%）
- 自己評価（10%）

**質問例：**
```
評価項目（1-5スケール）：
- 技術的専門性を発揮している
- チームワークを大切にしている
- 締切を守る
- 建設的なフィードバックを提供する
- 継続的に学習している

自由記述：
- 最も優れている点は？
- 改善すべき点は？
- 今後期待することは？
```

## まとめ

採用と評価は、エンジニアリングマネージャーの最も重要な責任の一つです。優秀な人材を獲得し、公正に評価し、継続的に成長させることで、強いエンジニアリングチームを構築できます。

採用においては、単にスキルだけでなく、文化適合性や成長可能性を評価することが重要です。また、採用プロセス自体が候補者体験となり、企業ブランドに影響することを忘れてはいけません。

評価においては、透明性と公正性が鍵となります。明確な基準、継続的なフィードバック、成長機会の提供により、メンバーのモチベーションと成長を促進できます。

最後に、採用と評価は継続的な改善が必要なプロセスです。データを収集し、フィードバックを受け、常により良い方法を模索し続けることが、長期的な成功につながります。

次章では、プロジェクトマネジメントについて詳しく見ていきます。優秀な人材を活かし、効果的にプロジェクトを遂行することで、ビジネス価値を創出します。
