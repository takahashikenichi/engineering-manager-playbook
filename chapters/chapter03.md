# 第3章：1on1とフィードバック - メンバーの成長支援

## はじめに - 1on1を設計しているか？

「1on1の30分にどれだけ力を注ぐかが、その週のメンバーの生産性を決めることになる」

以前、私のメンターだった上司から聞いた言葉にこのようなものがあります。「そんな大げさな」と最初は思ったものですが、EMとして数年経った今、この言葉は真実を示しているとも感じています。

1on1でメンバーの悩みや不満を早期に発見できれば、大きな問題になる前に解決できる。これは極めて当たり前のことですが、それがなかなかできません。メンバーのモチベーションの低下を察知して、手を打つ。それはメンバーの生産性を救い、チーム全体の生産性の保持へとつながるということはわかっているのですが。

そう。多くのマネージャーは、1on1を「進捗確認」や「タスクの割り振り」の場にしてしまっています。

ただ、これはある意味仕方のないこと、とも考えています。マネージャーは常に多忙で、メンバーの状況をつぶさに観察する余裕はあまりありません。週に1度の1on1があれば、その時間を使って積極的に**業務状況**を確認して次の手を打ちたいというのはマネージャーの本心でしょう。生産性を高めるといえば高めるのですが、それは極めて短視眼的ではあります。

本来の1on1は、メンバーを育て、マネージャーを育て、チームを育てるものに時間をあてなくてはなりません。進捗確認で終わらせてしまうと1on1の本質を見失ってしまいます。

本章では、そんな「組織を変えるための1on1」を考えます。

## 3.1 あらためて、1on1の本質と目的を考える

### 1on1とは何か - あなたの時間ではなく、部下の時間であるということ

「今週の1on1、何話そうかな...」

もしあなたがマネージャーで、メンバーとの1on1の前に毎週こう考えているなら、基本的な部分で1on1を理解できていないのかもしれません。1on1は**マネージャーの時間ではなくメンバーの時間**です。「進捗確認をしてしまう」というのはこの本質に反しています。

**1on1の本質：**
- メンバーのための時間であるということが大前提である
- 心理的に安全な対話の場である（メンバーは上司の機嫌を取らなくていい関係性であるべき）
- 継続的な関係構築のプロセス（1回の1on1で何か大きく変わることはないし、そのような期待もしない）
- 成長と発展の機会（長い時間軸で、たとえばキャリアなどついて考える貴重な時間にする）

**1on1でやってはいけないこと：**
- 「今週のタスクの進捗どう？」（それは別のミーティングで。仮に時間が無かったとしても、Slackで聞けば済ませられます）
- 「これやっといて」の一方的な指示（これも別のミーティングで。1on1を、仕事の指示をする場にはしない）
- 「君の評価はBだね」（評価面談は別で行いましょう）
- ただの雑談で30分（それは、まあ余裕があればそれもいいかもしれませんが、できれば別の機会に行いましょう）

### 1on1の目的 - 

**1. 早期問題発見 - 「最近元気ないな」を見逃さない**

1on1を行う理由の一つは、優秀なエンジニアが退職届を持ってきた時「全然気づかなかった」と愕然としないようにすることにあります。メンバーが自分自身のコンディションを自ら示す場として1on1は機能させるべきです。たとえば、会議での発言が減った、新しい技術への興味を示さなくなった、定時で帰るようになった...メンバーからのシグナルはいろいろあっても、1on1で話せる関係性を維持していれば早めにサインを出してもらます。

**2. 信頼関係の構築 - 「この人になら本音を言える」と思われる存在に**

「1. 早期問題発見」のために行うことでもありますが、信頼関係の構築の場として1on1を使いましょう。率直なフィードバックができる環境はチームの生産性を飛躍的に高めます*1。しかし、そのような「思っていることを素直に伝えあえる」関係性は一朝一夕には作れません。定期的な1on1はその積み重ねを通して「この人になら本音を言える」という信頼を生むのです。

**3. 個別最適化 - 「うまくいってる？」をサポートする**

1on1は1対1である、という当たり前のことではありますが、1対1であるが故にチーム全体への画一的なアプローチではなく、各メンバーの個性、強み、弱み、課題に応じたサポートをすることができます。特に弱みや課題について、フィードバックではフォローしきれないような場合は、1on1で中長期的なサポートを計画することができます。

**4. キャリア開発 - 「こうなりたい」をキャッチする**

チームで個人のキャリアについて話す場というのは考えにくいので、メンバーの長期的なキャリア目標について話し合う機会は、実質1on1以外には無いでしょう。これは前段の1〜3の項目の延長線上にあるものとも言えます。すなわち、メンバーのキャリアをマネージャーと話せる状況にあるためには、あらかじめ信頼関係を気かれており、問題を早期発見し個別に対処していくルーティンが確立されている必要はあります。そのルーティン自体が1on1の本質であり、ルーティンを確立するために1on1を使う、という形になります。

### 1on1を投資対効果で考える - 年間26時間の投資は高いか安いか？

しかし、マネージャーはいつも忙しい。メンバーのために、どこまでマネージャーの時間資源を1on1に投下すべきでしょうか？

メンバーひとり当たり週30分×52週＝年間26時間。大きな投資ですが、投資に見合う効果は得られるとすれば、下記の用に考えられるはずです。

- モチベーション低下による生産性損失：人件費の30-50%。短期的に見える損失としてはかなりの大きさです。
- エンジニア1人の採用コスト：300-500万円（実際はもっと掛かる可能性が高いです。仮に高額なエージェントを使わなくても、採用には人事やマネージャーの人件費がかかっているはずです。）
- 離職による知識の損失：ものによっては数千万円の価値を失うことになります。しかもその損失は不可逆的なものがほとんどです。
- 問題がエスカレートしてからの対処コスト：早期対処の10倍（と一般的には言われています）

こう考えると、週30分は最高にコスパの良い投資だと判断できますね。

## 3.2 効果的な1on1の構造

投資対効果が高いとはいえ、メンバー全員に週次で1on1をやる、というのはかなりの負担です。メンバー全員に満遍なく1on1をやるよりも、もっと効率の良いやり方はあるはずです。

### 1on1の頻度とスケジューリング

1on1の**推奨頻度** について考えてみましょう。マネージャーが直接面倒を見られるメンバーの人数の限界には諸説ありますが、大体10人くらいと言うと、あまり異論は出ないのではないでしょうか。では仮に、最大限のメンバー10人を抱えるマネージャーがいるといて、全員に毎週1on1をやるべきでしょうか？10人が30分ずつで週に5時間だけだから頑張ろう、と割り切ることもできますが、1on1の前後にはかならず付随する時間がでます。本来1on1においてマネージャー側は準備は不要ですが、1on1を通じて対応すべきことは多いはずです。これを考えると**全員に毎週1on1をする**とマネージャーのタイムマネジメントは相当難易度が高いものになるでしょう。

ではどうしたらよいのか？これは、ある程度明確な濃淡をつけて対応するしかありません。

経験則的には以下の配分がよいでしょう。
- 新入社員：週1回必須（最初の3ヶ月は絶対）
- ジュニアメンバー：隔週（成長期は頻度が大事）
- シニアメンバー：隔週〜月1回（でも本人が望めば毎週でも）
- 自律的なメンバー：月1回（ただし緊急時はすぐに時間を作る）

また、スケジューリングにも配慮する必要はあります。原則、定期的な繰り返し設定（同じ曜日・時間）に実施するようにして、可能な限りキャンセルはせずにリスケジュールするようにしましょう。また、午後の中途半端な時間に設定するなど、メンバーの生産的な時間を避けましょう。経験則的には、週の前半に1on1をしたほうが、その後のメンバーの行動にフォローアップの時間を確保しやすくなるので良いかと思います。

「忙しいから今週はスキップ」はマネージャーであればそれなりの頻度で起こりうることでしょう。本質的にはあまり良いことではありませんが、スキップするにしてもマネージャーからは可能な限りスキップせずにリスケジュールで対応するようにしましょう。スキップを続けると、メンバーは「自分は大切にされていない」と考えてしまい、1on1の本来の効果とは真逆になってしまいます。

### 1on1の環境 - 顔を合わせたほうがもちろん良い、のですが

リモートワーク・ハイブリッドワークが浸透している昨今、1on1を行う物理的な場所にそこまで拘る必要はないでしょう。ただ可能であれば、数回に1回はオフラインで開催することを提案しても良いかもしれません。

オフライン開催まではいかないまでも、リモート環境で1on1をする場合にはできるだけビデオオンにして顔だけは確認しあえるとよいと思います。ただ、ビデオオンは相手に強要すべきではありません。その上で、相手がビデオオンにしないのでこちらも、とはならずに、一方的にでもこちらの表情を知らせるようにしましょう。これもまた信頼関係の構築につながります。

### 時間配分とアジェンダ

すべてのミーティングは事前のアジェンダが重要であり、1on1もアジェンダを事前に用意することは1on1の質を決めるといっても過言ではありません。また、用意したアジェンダに対応した時間配分も重要です。1on1はメンバーがアジェンダを用意するべきであるため、マネージャー側で時間配分をマネージすることは本質からは外れるのですが、初回の1on1である程度型化された1on1時間配分を両者でして合意しておくと、以降の1on1で「時間が無くてあれを話せなかった」を減らすことができます。

例：30分の1on1の典型的な構成：

| 時間 | 内容 | 目的 |
|------|------|------|
| 0-5分 | チェックイン | 関係構築、雰囲気作り |
| 5-20分 | メンバーのトピック | メンバーの関心事項 |
| 20-25分 | マネージャーのトピック | フィードバック、情報共有 |
| 25-30分 | ネクストステップ | アクションアイテムの確認 |

例: 事前アジェンダのフォーマット

```markdown
# 1on1 ノート - [名前]

## 2024年3月15日

### メンバーのトピック
- プロジェクトXの技術的な課題について相談
- 来期のキャリア目標について
- チーム内のコミュニケーション改善提案

### マネージャーのトピック
- 先週のプレゼンテーションのフィードバック
- 新しいプロジェクトのアサインについて
- 四半期評価のスケジュール確認

### アクションアイテム
- [ ] [メンバー] 技術課題の解決案を文書化
- [ ] [マネージャー] メンタリング機会を探す
- [ ] [両者] 来週までにキャリア開発計画をレビュー

### 次回への持ち越し
- チーム合宿の企画について
```

前の章でも触れましたが、メンバーのトピックは進捗確認にならないように、マネージャーからのトピックは指示や評価にならないように気をつけましょう。

## 3.3 1on1で深堀りするための質問

私は1on1で沈黙したことはほぼないのですが、マネージャーでも1on1が苦手な人はいるものだと思います。一方メンバーにとっても、マネージャーとの時間はある種の緊張の場にはなりますので、話が弾まないこともあるでしょう。もしメンバー側からの情報が限定的だな、と感じたら、マネージャー側からうまく対話を深めるのもよいでしょう。以下に、1on1のトピックとして使える問い掛けを引用します。

**1. 現在の業務と課題**

- 「今週、一番ワクワクしたことは？」（ポジティブな問い掛けから）
- 「今一番モヤモヤしていることは？」（本音を引き出す）
- 「もし制約なくなんでもできる立場だったら、何を解決したい？」（制約を外してみる）
- 「最近、仕事で『ちょっと違うな』と思ったことは？」（改善ポイントを探る）

**2. キャリアと成長 - 「5年後の夢」も「3ヶ月後の現実」も**

- 「3ヶ月後、どんなスキルを身につけていたいですか？」（具体的・現実的。目標設定とも関係）
- 「社内で『この人すごい』と思う人は誰ですか？なぜですか？」（身近なロールモデル）
- 「最近学んだことで、一番『使える』と思ったものは何ですか？」（直近の興味や考えのクセを知る）

キャリアの話は壮大すぎると空虚になってしまいますので、短い時間軸と長い時間軸でのキャリアをうまく分けて考えるとよいでしょう。

**3. チームとの関係 - 「仲良し」の裏**

- 「チームで一番頼りにしている人は誰ですか？」（信頼関係）
- 「最近、誰かに助けられたこと、助けたことはありますか？」（相互支援）
- 「もし新しいチームのリーダーになったとして、最初に誰を誘いますか？」（メンバー同士の関係性）
- 「チームで『これ、誰も言わないな...』と思うことはありますか？」（未解決の問題）

### 深掘り質問のテクニック

**あらかじめ決められた選択肢や回答を持たない質問をする**

悪い例と良い例：
- 悪い例：「プロジェクトは順調ですか？」→ 「はい」で終了・大体順調だと答えます
- 良い例：「プロジェクトの中で、今一番注目していることは何ですか？」→ 話が広がります

**「なるほど」のあとのフォローアップをする質問をする**

- 「なるほど。たとえば具体的には何がありますか？」（具体例を引き出す）
- サンドイッチアプローチ
  1. まずポジティブから：「○○の部分は本当に素晴らしかった」
  2. 次に改善点：「一方で、もっと良くできたかもしれない部分がある」
  3. 最後にポジティブで締める：「でも全体的には成長しているし、次はもっとできる」
- 「最近、新しいチャレンジが少ないように見えますが、もっと成長できる機会を作るには何が必要でしょうか？」→ 共に解決策を探る姿勢を示すことができます。

## 3.4 フィードバックの技術

### 「評価の段階で初めてフィードバックを聴きました」

ここまで1on1の重要性と手法について説明してきましたが、1on1よりも頻度の高いコミュニケーションにフィードバックがあります。フィードバックというと、その期の最後に評価説明として行われるものを想定されるかもしれません。しかし、「目標の達成や評価の段階でフィードバックをする」ことは、メンバーが改善して評価を戻す（上げる）機会を奪ってしまうことになり最悪です。

**つまり、フィードバックとは可能な限り短いサイクル行われる必要があります。**

ただ、マネージャーからすれば、日々の業務に追われていたとか、タイミングを見計らっていたとか、フィードバックができない言い訳も出てくるでしょう。しかしそれは、本質的にフィードバックを伝えることから逃げていた、という要素はなかったでしょうか。言いにくいことを先延ばしにした結果、半年分の改善機会をそのメンバーから奪ってしまう。メンバーはモチベーションが下がり不信感をつのらせます。たしかに最悪です。

フィードバックは「その場で、すぐに」が鉄則です。1on1はフィードバックの場としては適切ではありません。言い回しを考えている間に、フィードバックの鮮度は失われます。多少ぎこちなくても、早く伝えたほうがずっと良い。これはそういった場面に遭遇した回数が増えるほど、正しいと確信しています。

### SBIモデル

とはいえ、「すぐに伝える」と「感情的に伝える」は本当に紙一重です。特に改善を求めるフィードバックは、伝え方を間違えるとせっかくの関係性を壊しかねません。

こういった状況の中で頼りになるのは、Center for Creative Leadershipが開発した**SBIモデル**です。

- Situation（状況）： いつ、どこで起きたか
- Behavior（行動）： 具体的にどのような行動が観察されたか
- Impact（影響）： その行動がもたらした影響は何か

この3つの要素でフィードバックを組み立てる、というシンプルなフレームワークなのですが、これがあるだけで曖昧で攻撃的な表現を避けることができるでしょう。

たとえば、

> 昨日のアーキテクチャレビューで（状況）、詳細な技術文書と図解を準備して説明してくれたおかげで（行動）、チーム全員が複雑なシステムを理解でき、建設的な議論ができました（影響）。

といったポジティブなフィードバックも


> 今朝のスタンドアップで（状況）、進捗報告が技術的な詳細に入りすぎていたので（行動）、他のメンバーの共有時間が削られてしまいました（影響）

といったネガティブなフィードバックも、どちらもSBIで伝えると「何が良かったのか・何が悪かったのか」が具体的になります。これは単に「良かった・駄目」と言うよりもはるかにメンバーの次の行動につながります。ネガティブなフィードバックもSBIで伝えれば、人格の否定ではなく特定の行動についての話だと相手も理解しやすくなります。

マネージャーでなくとも、フィードバックが得意、という人は少ないと思います。そういうときこそ、フレームワークという構造に頼ることには意味があると思っています。慣れてくれば意識しなくても自然とこの形で話せるようになります。大事なのはフレームワークを忠実に再現することではなく、具体的な事実に基づいて伝えるという姿勢のほうです。

### フィードバックをどう使い分けるか

フィードバックの種類は大きく分けて３つあります。

- 良い行動を認めるもの
- 行動の改善を求めるもの
- そしてメンバーの成長を促すもの

これらはフィードバックの目的も行うタイミングも異なるのですが、実際のフィードバックの多くは「改善を求めるフィードバック」に偏りがちです。

良い行動を認めるフィードバックは、できるだけその場で、できれば他のメンバーの前で伝えるのが効果的です。ただし、人前で褒められることを好まないメンバーもいますので、本人の性格や好みは事前に把握しておく必要があります。「今日のコードレビューでの建設的なコメント、ジュニアメンバーの学習に大きく貢献していましたよ」という一言が、思った以上にメンバーの行動を強化します。

一方で、行動の改善を求めるフィードバックは必ず1対1の場で行います。これは鉄則です。人前で指摘された記憶は、内容に関係なく屈辱として残ります。伝える側としては「軽い指摘のつもり」でも、受ける側にとってはそうではないことがほとんどです。

意外と見落としがちなのが、成長を促すフィードバックです。これは問題があるから伝えるのではなく、メンバーのキャリア目標に紐づけて「次の一歩」を示すものです。「プレゼンテーションスキルを伸ばしたいと言っていたので、来月の全社会議でチームの成果を発表する機会を作りました」のように、フィードバックと機会提供をセットにすると、メンバーにとっては言葉だけでない具体的な支援として受け取れます。

### フィードバックを受け取る側の話

フィードバックの「伝え方」については多くのことが語られていますが、「受け取り方」についてはあまり議論されていないように思います。しかし、本当はフィードバックが機能するかどうかは、伝える側だけでなく受け取る側の姿勢にもかかっています。

防御的にならず耳を傾ける、というのは簡単ではありません。特にネガティブなフィードバックを受けた直後は、反射的に「でも、それは...」と言いたくなるものです。私自身もそうです。フィードバックを受ける側の心構えとして覚えておいてほしいことは以下になります。

- 防御的にならない
- 明確化のための質問をする
- 感謝を示す
- 行動計画を立てる
- フォローアップする

フィードバックに対して必要以上に心を揺さぶられない（揺さぶらない）ためにも、事前に「フィードバックにはまず感謝を伝え、時間を置いてから反応する」ことをメンバーに伝え、マネージャー自らも心がけるようにしましょう。その場で完璧に受け止める必要はなく一度持ち帰って咀嚼する時間があっていい、そのうえで、わからない点があれば質問し、自分なりの改善の方向性を考えてくれれば十分です。

こうした「フィードバックの受け取り方」は、あらかじめチーム内で共有しておくと、フィードバック自体へのハードルが双方にとって下がります。伝える側も「ちゃんと受け止めてもらえる」という安心感があるから、より率直に伝えられるようになる。この循環が生まれると、フィードバックがチームの文化として根付いていきます。

## 3.5 異なるタイプのメンバーにどう対応するか？

### メンバーのパーソナリティに合わせたコミュニケーション

1on1の進め方に唯一の正解はありません。当たり前のことですが、メンバー全員に同じアプローチで臨むのは、うまくいかないことのほうが多いでしょう。

私のチームにいた優秀なエンジニアは、会議では一言も発しない人でした。最初は「消極的なのかな」と思ったこともあります。しかし、1on1では素晴らしいアイデアを次々と出してくれました。彼に必要だったのは「考える時間」だったのです。事前にアジェンダを共有して、翌日の1on1で話してもらう。Slackで「これについてどう思いますか？明日聞かせてください」と一言添える。それだけで、彼の1on1は驚くほど充実したものになりました。沈黙を「気まずいもの」として埋めようとせず、相手が考えている時間を尊重する。内向的なメンバーとの1on1では、この姿勢が何よりも重要です。

逆に、話し始めたら止まらないメンバーもいます。これはこれで素晴らしい資質なのですが、1on1の時間が発散しやすいという側面もあります。こういうメンバーには、ホワイトボードや画面共有で話を視覚化しながら進めると、議論が整理されやすくなります。「それは面白いですね。もっと聞かせてください」と深掘りしつつ、最後にはアクションに落とし込む。エネルギーを方向づけることがマネージャーの役割になります。

「まだ70%なので...」と言い続ける完璧主義的なメンバーには、「完璧じゃなくていい」と伝えても響きません。それよりも「ここまでは完璧に、ここからは70%でいい」と線引きを一緒にするほうが、安心して前に進めるようです。リスクを一緒に洗い出して、「最悪こうなっても大丈夫」という見通しを共有するだけで、行動が早くなることがあります。

一方で、大きなビジョンを語るのが好きなメンバーもいます。壮大なアイデアは素晴らしいのですが、ときどき地に足がつかなくなります。こういう場合は、まず大きな絵を描ききってもらって、それから「それを実現するための最初の一歩は何ですか？」と問いかける。否定から入るのではなく、実現に向けた具体化を一緒に考えるという姿勢が大事です。

### キャリアステージごとに1on1の意味は変わる

メンバーの経験レベルによって、1on1で求められるマネージャーの役割はかなり異なります。経験年数だけで一律に判断するのは危険ですが、それでもある程度の傾向は存在します。

新卒やジュニアのエンジニアに対しては、「教える」よりも「一緒に学ぶ」というスタンスが有効だと感じています。以前、新卒のメンバーに「gitのrebaseは使えますか？」と聞いたことがあります。「はい」と答えてくれたのですが、実は使ったことがなかった。結果、後日大混乱になりました。ここから学んだのは、知らないことを「知らない」と言える環境づくりが最優先だということです。週1回の1on1を通じて「今週一番難しかったことは何ですか？」と聞き続けるだけで、少しずつその壁は下がっていきます。「わからないことリスト」を一緒に作るのも効果的でした。

3〜5年目のミッドレベルのエンジニアは、実は一番難しい時期にいます。周囲からは「もう一人前」と見なされていますが、本人はまだ不安を抱えていることが多い。この段階では、答えを教えるのではなく「どう思いますか？」「あなたならどうしますか？」と繰り返し問いかけることを意識しています。答えをすぐに渡さない、というのはマネージャー側にも覚悟がいることです。リカバリーできる範囲であえて失敗させることも含めて、メンバーの自走力を育てるフェーズだと捉えています。1on1では「今のプロジェクトで、一番『これは自分がやった』と言えることは何ですか？」という問いが、オーナーシップの意識を育てるのに役立ちました。

シニアレベルになると、1on1の性質は「管理」から「作戦会議」に変わります。「正直、今のアーキテクチャには限界を感じています」「そうですね。では、どうしましょうか？」「3ヶ月かけて段階的に移行したいです」「予算を確保しますので、提案書を作っていただけますか？」。これは実際にあった会話ですが、マネージャーとメンバーというよりも、対等なパートナーとして技術的な課題に向き合う関係です。シニアエンジニアに対して技術的な知ったかぶりをすると見透かされますし、そもそも技術力で対抗する必要はありません。むしろ、組織の文脈や経営層の考えなど、マネージャーだからこそ持っている情報を共有することのほうが、シニアにとっては価値があります。

スタッフやプリンシパルレベルのエンジニアとの1on1は、正直、私が学ぶことのほうが多いです。「今回の障害、根本原因は組織構造にあると思います」という一言から、2時間の議論を経て組織改革のアイデアが生まれたこともあります。このレベルの人には1on1を押し付けるのではなく、本人が必要な時に場を設けるという柔軟さが必要です。「教えてほしいことがある」というスタンスで臨むほうが、お互いにとって実りのある時間になります。

### パフォーマンスの違いにどう向き合うか

パフォーマンスの高低でメンバーを分類すること自体に抵抗を感じるかもしれません。しかし、現実として、メンバーのパフォーマンスレベルによって1on1で注意すべきポイントは異なります。

「あの人は大丈夫」が一番危険な判断です。ハイパフォーマーは優秀であるがゆえに仕事が集中しやすく、本人も限界まで頑張ってしまう傾向があります。あるエースエンジニアに仕事が集中し、ある日突然「もう無理です」と言われたことがあります。3ヶ月の休職、チームは大混乱。完全に私の責任でした。それ以来、ハイパフォーマーには「最近、楽しいですか？」とパフォーマンスではなく感情を聞くようにしています。必要に応じて仕事を断る練習をさせたり、長めの休暇を推奨したりと、意識的にペースを調整する関わりが必要です。

チームの多数を占めるのは、安定して成果を出し続けているメンバーです。「普通」と括ってしまうのは適切ではなく、こうしたメンバーこそチームの基盤を支えています。1on1では、小さな成功を具体的に認めることが効果的です。「このバグ修正、本当に助かりました」という一言は些細に見えますが、「自分の仕事はちゃんと見てもらえている」という実感につながります。得意分野を見つけて任せる、勉強会での発表を勧めるなど、少しずつ活躍の場を広げる支援をしていくと、本人の自信も変わってきます。

パフォーマンスが低いメンバーに対しては、「ダメだ」というレッテルを貼る前に立ち止まって考える必要があります。パフォーマンスが低いことと、能力がないことはイコールではありません。プライベートで大変なことが起きているのかもしれない。タスクが合っていないのかもしれない。そもそも、何を期待されているのか理解できていないのかもしれない。ミスを恐れて動けなくなっている可能性もあります。大事なのは、原因を一緒に探ることです。「最近調子が良くないように見えますが、何か困っていることはありますか？」。この一言から始めることで、表面的なパフォーマンスの問題の裏にある本当の課題が見えてくることがあります。

## 3.6 キャリア開発の支援

### キャリア会話の進め方 - 「夢」の話だけでは不十分

「5年後の自分を想像してください」というのは、正直難しいものです。IT業界で5年後を予測することは、誰にもできません。

**もっと現実的なキャリア対話の切り口：**

私が実際に使っている質問：
- 「最近、仕事で『これは楽しい』と思った瞬間はありますか？」（情熱の源泉を探る）
- 「今の会社で『この人はすごい』と思う人は誰ですか？なぜですか？」（具体的なロールモデル）
- 「もし明日から何でも挑戦できるとしたら、何をしますか？」（制約を外した本音）
- 「正直、何が一番不安ですか？」（キャリア不安は誰にでもあります）

**現実的なギャップ分析 - 「今できること」から始める：**

ある部下との実際の会話：
「テックリードになりたいです」
「今すぐできることは何ですか？」
「...コードレビューをもっと丁寧に行うことでしょうか」
「いいですね。明日から誰のコードレビューをしますか？」

大きな目標も、明日からできる小さな一歩に分解する。これが秘訣です。

### 個人開発計画（IDP）の作成 - 実効性のある計画の作り方

IDPを作って満足していませんか？私も昔はそうでした。年度初めに立派な計画を作って、次に見るのは年度末の評価時でした。

**生きたIDPの作り方：**

実際に機能したIDPの例：
```markdown
# ○○さんの成長計画（2024年度）

## 今年の挑戦したいこと（1つだけ！）
「技術的な議論でもっと発言できるようになる」

## 3ヶ月後の具体的な姿
- 設計レビューで必ず1回は質問する
- 自分の意見を図解して説明できる
- 「わからない」を恥ずかしがらない

## 今月やること（超具体的に）
□ 毎週水曜の設計レビューに参加
□ わからない用語をメモして調べる
□ 先輩の発表スタイルを観察する

## 必要なサポート
- レビュー前に資料を見せてもらう
- 質問していいタイミングを教えて
- 月1で練習に付き合って
```

ポイント：
- 目標は1つに絞る（欲張ると何も達成できない）
- 3ヶ月単位で見直す（1年は長すぎる）
- 「今月」のアクションを明確に（明日から何するか）

### ストレッチアサインメント - 適切な挑戦レベルの見極め方

部下に難しい仕事を任せて失敗させた経験はありませんか？私はあります。「成長のため」と言いながら、実は丸投げでした。

**失敗から学んだ「70-20-10ルール」：**

実際の適用例：
「新しいAPIを作っていただきたいのですが」
- 70%（できること）：基本的なCRUD処理の実装
- 20%（背伸び）：認証機能の実装
- 10%（チャレンジ）：パフォーマンスチューニング

これなら、失敗しても立ち直れます。全部が新しいと、心が折れてしまいます。

**実際に効果があったストレッチ機会：**
- 金曜午後の「触ってみようタイム」（新技術を2時間だけ試す）
- 他チームへの「留学」（1週間だけ別チームで働く）
- 社内LT大会での「5分間登壇」（いきなり外部はハードル高い）
- バグ対応の「ペア作業」（一人じゃないから安心）
- 議事録係から始める「会議参加」（聞くことから始める）

### メンタリングとスポンサーシップ - 「師匠探し」だけでは不十分

「メンターを見つけました」で終わっていませんか？

**メンタリングが失敗する理由：**

実際の失敗例：
「○○さん、技術力が高いので、あなたのメンターをお願いしました」
→ 2回会って自然消滅

何が悪かったか：
- 相性を考えていない（技術力≠良いメンター）
- 目的が不明確（何を学ぶの？）
- 時間の確保なし（忙しい人同士...）
- フォローアップなし（放置プレイ）

**うまくいったメンタリングの例：**

「○○さんとランチに行きませんか？実は彼も昔、同じことで悩んでいました」
→ 自然な出会いから始める
→ 共通の悩みでつながる
→ 定期的なランチ会に発展
→ 自然なメンター関係が生まれる

**スポンサーシップの本質：**

メンタリング：「こうやるといいよ」（アドバイス）
スポンサーシップ：「この人すごいんですよ！」（機会を作る）

私がやっていること：
- 会議で部下の成果を代弁（「これは○○さんのアイデアです」）
- 上司との1on1で推薦（「○○さん、次のリーダー候補です」）
- 社外発表の機会を作る（「うちの○○に話させていただけませんか？」）

## 3.7 困難な会話の進め方 - 「言いにくいこと」を伝える責任

### 準備の重要性 - 準備不足がもたらす問題

困難な会話を先延ばしにした結果、チームが崩壊した経験があります。「まあ、そのうち改善するだろう」は甘い考えでした。

**本当に必要な事前準備：**

実際にやってよかったこと：
- 具体的な事例を3つ用意（「いつも」「みんな」は禁句）
- 言いたいことを箇条書きにする（感情的にならないため）
- タイミングを計る（金曜の夕方は最悪。月曜の午前中がベスト）
- 部屋を予約する（会議室は圧迫感があるので、カフェスペースも検討）
- 想定問答を作る（「それは誤解です」への返答を準備）
- 次のアクションを決めておく（話して終わりは無意味）

### パフォーマンス問題への対処 - 問題の早期対処の重要性

**段階的アプローチ - 実際の会話例で学ぶ：**

**第1段階：まず相手の認識を確認する**

私の失敗例：
「あなたのパフォーマンスが落ちています」
→ 即座に防御的になり、建設的な会話ができなかった

成功例：
「最近のプロジェクトで少し苦戦しているように見えますが、どう感じていますか？」
→ 「実は...」と本音を話してくれた

**第2段階：期待値をすり合わせる（押し付けない）**

悪い例：
「ルールだからやってください」

良い例：
「24時間以内のレビューは難しいですか？実は他のメンバーが待っていることをご存知でしたか？」
→ 影響を理解してもらう

**第3段階：本当の原因を一緒に探る**

表面的な質問：
「なぜできないのですか？」

深掘りする質問：
「レビューの時、一番何に時間がかかっていますか？」
「他にもっと優先度の高いタスクがありますか？」
「実は何か困っていることがありますか？」

**第4段階：一緒に解決策を作る（押し付けない）**

私がよく使うフレーズ：
「では、このような方法はいかがでしょうか？」
「試しに2週間やってみて、ダメなら別の方法考えよう」
「小さく始めて、徐々に増やしていこう」

具体例：
- 朝一の30分だけレビュータイム
- 複雑なPRは一緒にペアレビュー
- 週1回振り返りをして調整

### 感情的な状況への対処 - 感情への適切な対応

**感情的な反応への実践的対応：**

**怒りへの対処 - 火に油を注がない：**

実際にあった場面：
部下「なぜ私だけ言われるのですか」
私（深呼吸して）「怒るのは当然だと思います。5分休憩して、コーヒーでも飲んでから続きを話しましょう」
→ 一旦クールダウンすることで建設的な会話ができた

やってはいけないこと：
- 「落ち着いて」と言う（火に油）
- 言い返す（エスカレート）
- 無視する（関係性悪化）

**涙への対処 - 泣くことは悪いことじゃない：**

私も部下の前で泣かれたことがあります。最初はパニックでした。

学んだこと：
- そっとティッシュを渡す（何も言わない）
- 「ゆっくりでいいよ」と伝える
- 無理に励まさない
- 「今日はここまでにして、明日続きを話しましょうか」と提案する

**否定への対処 - 「そんなことない！」を受け止める：**

「私はちゃんとやっています」
「そう感じているのですね。しかし、具体的にこういうことがありました」
→ 感情を否定せず、事実を伝える

### 終了・退職の会話 - 最後まで敬意を持って対応する

**自主退職の場合 - 「裏切られた」と思わない：**

優秀な部下が辞めた時、正直ショックでした。しかし、それは彼らの人生です。

私が心がけていること：
1. まず「おめでとう」と言う（新しい挑戦を祝福）
2. 理由を素直に聞く（組織改善のヒント）
3. 無理な引き止めはしない（関係性が壊れる）
4. 最高の送り出しをする（いつか一緒に働くかも）
5. LinkedInでつながり続ける（人脈は財産）

**解雇の場合 - 人生で最も辛い仕事：**

解雇を伝えた経験があります。今でも胸が痛みます。

絶対に守るべきこと：
- 事前にHRと入念に準備（法的リスクを避ける）
- 簡潔に、しかし人間的に伝える
- 「あなたが悪い」ではなく「役割が合わなかった」
- プライバシーを守る（他のメンバーに詳細を話さない）
- 最後まで敬意を持って接する

## 3.8 リモート環境での1on1 - 画面越しでも心を通わせる方法

### リモート特有の課題 - オンライン疲労の原因と対策

コロナ禍で全員リモートになった時、1on1の質が大きく低下しました。なぜでしょうか？

**私が気づいた本当の課題：**

| 課題 | よくある対策 | 本当に効果があった対策 |
|------|------------|--------------------|
| 表情が読めない | カメラON必須 | 「今日はカメラOFFでもいいよ」の選択肢 |
| 接続トラブル | 事前テスト | 「つながらなかったら電話」のルール |
| 集中できない | 静かな場所 | 「歩きながら話そう」も許可 |
| 雑談がない | アイスブレイク | 最初の5分は仕事の話禁止 |
| 時差 | 交互に調整 | 「朝型/夜型」を聞いて固定時間設定 |

### バーチャル1on1のベストプラクティス - リモートでも質の高い対話を実現する

**環境設定 - 完璧を求めすぎない：**

私の失敗：
「静かな環境でお願いします」と言い続けた結果、子育て中のメンバーが1on1を避けるように...

今のルール：
- 子供の声が入っても問題ありません（むしろ家族の話題で盛り上がります）
- 散歩しながらでも構いません（リラックスして本音が出やすくなります）
- カフェからでも構いません（環境を変えると気分も変わります）
- パジャマでも構いません（朝一の1on1の場合）
- 画面共有できればどこでもOK

**エンゲージメントの工夫 - 相手の注意を引きつける方法：**

効果があったテクニック：
- 名前を呼ぶ（「〇〇さん、それはつまり...」）
- 相槌を大きめに（画面越しは通常の2倍程度が適切です）
- チャットも併用（「今の話、とてもよく分かります」）
- 画面共有で図を描きながら話す
- たまに質問を投げる（「ここまでで質問はありますか？」より「どう思いますか？」）

### 非同期要素の活用 - 限られた時間を最大限に活用する

**事前準備で時間を節約：**

実際に使っているSlackメッセージ：
```
明日の1on1の準備
1. 話したいことがあれば教えてください（なければこちらから話題提供します）
2. 最近の気分を一言で（良い・普通・悪いなど）
3. 緊急で話したいことはありますか？（あれば最優先で対応します）
```

これだけで、1on1の最初の10分が節約できます。

**フォローアップも非同期で：**

1on1後、必ずやること：
- その場でアクションをTrelloに登録（忘れ防止）
- 重要な決定事項をSlackに投稿（認識合わせ）
- 参考URLを共有（後で見てもらう）
- 次回の宿題を明文化（曖昧にしない）

「議事録」という堅苦しいものではなく、箇条書きで十分です。

## 3.9 1on1の効果測定と改善

### 効果測定の指標

**定量的指標：**
- 1on1の実施率
- キャンセル/リスケジュール率
- アクションアイテムの完了率
- メンバーの満足度スコア
- 離職率の変化

**定性的指標：**
- 関係性の深さ
- 問題の早期発見数
- キャリア成長の実感
- チームの雰囲気
- フィードバックの質

### メンバーからのフィードバック

**定期的なサーベイ：**
```
質問例：
1. 1on1は価値がありますか？（1-5スケール）
2. 頻度は適切ですか？
3. あなたの成長に貢献していますか？
4. 安心して話せる環境ですか？
5. 改善してほしい点はありますか？
```

### 継続的な改善

**自己振り返りの質問：**
- 今日の1on1で最も価値があった部分は？
- メンバーは十分に話せたか？
- 次回フォローアップすべきことは？
- 違うアプローチを試すべき点は？

**スキル向上の方法：**
- 他のマネージャーとのシャドーイング
- コーチングトレーニング
- フィードバック練習
- ロールプレイ
- 外部研修への参加

## まとめ

1on1とフィードバックは、エンジニアリングマネージャーの最も重要なツールの一つです。これらを効果的に活用することで、メンバーの成長を促進し、チームのパフォーマンスを向上させ、強い信頼関係を構築することができます。

重要なのは、1on1を単なる定例会議としてではなく、メンバーへの投資として捉えることです。一人一人に合わせたアプローチを取り、継続的に改善を重ねることで、1on1の価値は飛躍的に増大します。

最後に、完璧な1on1というものは存在しません。メンバーとの関係性、状況、文化によって、最適なアプローチは異なります。本章で紹介したフレームワークやテクニックを出発点として、あなた自身のスタイルを確立してください。

次章では、採用と評価について詳しく見ていきます。優秀な人材を獲得し、公正に評価することは、強いチームを構築し維持するための基盤となります。
